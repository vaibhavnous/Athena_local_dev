Here is the high-resolution, slightly more detailed breakdown of the Athena Pipeline. This version is designed to give you and your team a granular understanding of exactly how data flows and where the "moving parts" live.🪨 Athena Pipeline: Technical Architecture1. Data Ingestion & Transformation (The "Bronze" Layer)Source Input: Handles .txt, .docx, or raw strings. Normalized into brd_text.Identity Logic: Every BRD gets a SHA-256 fingerprint. This is the primary key for the entire system's memory.Cost Control: Initial tiktoken estimate happens here. If $>50k$ tokens, the pipeline hard-fails before spending any API budget.Vectorization:BRD: Chunked and sent to Pinecone namespace global.Schema: INFORMATION_SCHEMA columns are converted to "semantic sentences" and sent to Pinecone namespace schema.2. Dual-Layer Memory (The Efficiency Engine)Layer 1 (Exact Match): If the fingerprint exists in Azure SQL ai_store, the pipeline hydrates (auto-fills) all downstream state and moves to the final steps.Layer 2 (Semantic Match): If no exact match, it queries Pinecone.Score > 0.75: Reuses historical "Context KPIs."Score < 0.3: Uses these as "Rejected KPIs" (negative memory) to ensure the LLM doesn't repeat past mistakes.3. Orchestration Flow (LangGraph + HITL)The pipeline moves through 9 Nodes defined in graph.py.NodeActionShared State OutputIngestionParse/Embedrun_id, fingerprint, brd_textMemoryLookupmemory_layer1/2, prior_kpisReq ExtractLLM Structuringreq_business_objective, req_constraintsKPI ExtractGroundingextracted_kpisGate 1 (HITL)Human Reviewcertified_kpis, human_decisionNominationSearch/Expansionnominated_tables, semantic_matchesGate 2 (HITL)Human Reviewcertified_tablesDiscoverySQL Crawldiscovered_metadata (Types, Nullability)ProfilingSQL Pushdowncolumn_profiles (Percentiles, Cardinality)4. Search & Nomination Heuristics (Node 6)This is the most complex node. It doesn't just "search"—it fuses four methods:Lexical: Matches tokens against table/column names (snake_case/camelCase aware).Semantic: Pulls from Pinecone metadata index.FK Expansion: Uses REFERENTIAL_CONSTRAINTS to find related tables that might hold missing data.Lookup Sweep: Explicitly looks for reference data (dim_, lkp_, ref_).5. Persistence Layer (Azure SQL)brd_run_registry: Tracks every execution.ai_store: Stores the actual JSON payloads (Requirements, KPIs, etc.).hitl_review_queue: Manages the task list for the human operator.kpi_checkpoints: Stores the full Stage01State so the pipeline can be resumed after a pause.
